# 性能指标小白学习文档

> 项目：desktop-pet
> 适用范围：桌宠 / AI 伴侣产品的 PC 和移动端性能测试
> 目标读者：第一次看 CPU、GPU、内存、FPS、能耗等指标的人
> 最后更新：2026-05-08

## 1. 先建立一个直觉

性能测试不是只看“CPU 高不高”。对桌宠来说，真正要回答的是：

1. 它会不会拖慢电脑或手机？
2. 它会不会影响游戏 FPS？
3. 它会不会一直占着内存或显存？
4. 它会不会让风扇、发热、耗电变明显？
5. 它会不会抢输入、抢麦克风、请求敏感权限？
6. 它的资源占用是偶尔突发，还是一直居高不下？

最简单的判断方式：

| 项目 | 看什么 | 小白理解 |
| ---- | ---- | ---- |
| CPU | 平均、P95、峰值 | 大脑忙不忙 |
| GPU | 平均、P95、峰值 | 显卡忙不忙 |
| 内存 | 平均、峰值、是否持续上涨 | 占了多少工作空间 |
| GPU 内存 | 平均、峰值 | 占了多少显卡工作空间 |
| FPS | 开桌宠前后变化 | 游戏是否变卡 |
| 网络 | 流量、请求频率 | 是否一直联网 |
| 能耗 / 发热 | 电量、功耗、温度、风扇 | 是否更耗电更热 |
| 体验 | 卡顿、输入冲突、权限打扰 | 用户是否感觉被打扰 |

## 2. CSV 基础字段

这些字段不是性能本身，但决定数据能不能被相信。

| 项目 | 小白解释 | 为什么重要 | 填写建议 |
| ---- | ---- | ---- | ---- |
| 产品 | 被测试的产品名称 | 用于横向对比 | 保留原产品名，例如 Desktop Mate、VPet-Simulator |
| 平台 | 产品运行的平台 | 同一产品在不同平台表现可能完全不同 | Windows、macOS、iOS、Android |
| 产品版本 | 当前测试的应用版本 | 版本变化可能改变性能 | 尽量记录完整版本号 |
| 测试设备 | 运行测试的设备 | 设备越强，指标越容易好看 | 写 CPU、GPU、内存、机型 |
| 系统版本 | 操作系统版本 | 系统调度和驱动会影响结果 | Windows 11、macOS 版本、iOS / Android 版本 |
| 场景 | 正在测试什么行为 | 同一个产品在不同场景占用差异很大 | 冷启动、可见空闲、语音聊天、全屏游戏覆盖层 |
| 时长秒 | 场景运行了多久 | 测太短容易只看到偶然波动 | 空闲建议 600 秒，游戏建议 900 秒 |
| 测试人 | 谁测的 | 方便回溯和复查 | 写名字或代号 |
| 日期 | 测试日期 | 性能结果会随版本和设备变化 | 使用 YYYY-MM-DD |
| 置信度 | 对这条数据有多确定 | 区分粗测和严格测试 | 高、中、低 |
| 备注 | 额外说明 | 记录异常、前提、未覆盖内容 | 写清楚限制条件 |

## 3. CPU 指标

CPU 可以理解为设备的“通用计算大脑”。AI 文本处理、音频处理、窗口管理、脚本逻辑、网络回调等都会用 CPU。

| 项目 | 单位 | 小白解释 | 怎么看 |
| ---- | ---- | ---- | ---- |
| CPU平均占用_pct | 百分比 | 整个测试期间平均用了多少 CPU | 越低越好，代表长期负担小 |
| CPU_P95占用_pct | 百分比 | 95% 的时间里 CPU 不超过这个值 | 比平均值更能反映“偶尔卡一下”的风险 |
| CPU峰值_pct | 百分比 | 测试中最高的一瞬间 CPU 占用 | 用来发现启动、语音、截图等突发开销 |

### 3.1 平均值、P95、峰值的区别

| 项目 | 小白比喻 | 适合回答的问题 |
| ---- | ---- | ---- |
| 平均值 | 一整天平均有多忙 | 这个产品长期费不费资源 |
| P95 | 大多数时候最糟到什么程度 | 用户是否会经常感觉卡 |
| 峰值 | 最忙的一瞬间 | 是否有明显突刺或卡顿风险 |

例子：

| 项目 | CPU P95 | CPU 峰值 | 解读 |
| ---- | ---- | ---- | ---- |
| 1% | 3% | 20% | 平时很轻，只是偶尔突发 |
| 8% | 12% | 20% | 长期偏重，空闲也可能耗电 |
| 1% | 15% | 80% | 平均好看，但体验可能有卡顿尖峰 |

### 3.2 对桌宠的判断

| 项目 | 比较理想 | 需要警惕 |
| ---- | ---- | ---- |
| 2D 空闲桌宠 | 平均 <= 1%，P95 <= 3% | 空闲长期超过 5% |
| 3D / Live2D 空闲 | 平均 <= 2%，P95 <= 5% | 空闲长期接近 10% |
| 语音聊天 | 可以短时升高 | 长时间维持高 CPU |
| 本地 AI | 允许高占用，但必须显式开启 | 默认开启导致游戏卡顿 |

## 4. GPU 指标

GPU 可以理解为设备的“图形计算大脑”。3D 角色、Live2D、透明窗口、动画、WebGPU、本地视觉模型都可能用 GPU。

| 项目 | 单位 | 小白解释 | 怎么看 |
| ---- | ---- | ---- | ---- |
| GPU平均占用_pct | 百分比 | 测试期间 GPU 平均忙碌程度 | 越低越不影响游戏和电池 |
| GPU_P95占用_pct | 百分比 | 大多数时间里 GPU 的高位水平 | 用来看动画是否稳定吃 GPU |
| GPU峰值_pct | 百分比 | GPU 最高的一瞬间占用 | 用来看启动、切换、截图、3D 动作突发 |

### 4.1 GPU 为什么对桌宠特别重要

桌宠经常是一个“浮在桌面上的窗口”。它可能和游戏、直播工具、浏览器、桌面合成器一起抢 GPU。

| 项目 | 说明 |
| ---- | ---- |
| 2D 静态贴图 | 通常低 |
| 2D 高频动画 | 中低 |
| Live2D | 中 |
| 3D VRM 角色 | 中高 |
| WebGPU / 本地视觉 | 高 |
| 全屏游戏覆盖层 | 需要重点测 |

### 4.2 GPU 指标怎么解读

| 项目 | 可能原因 | 处理方向 |
| ---- | ---- | ---- |
| 空闲 GPU 长期高 | 动画帧率过高、透明窗口合成成本高 | 降低 FPS、静止时暂停动画 |
| 峰值很高但很短 | 启动加载、模型切换、截图分析 | 可以接受，但要避免卡住游戏 |
| 游戏 FPS 下降明显 | 与游戏抢 GPU、显存或合成器资源 | 加游戏模式、隐藏策略、降帧 |

## 5. 内存指标

内存可以理解为应用占用的“桌面空间”。内存越高，系统留给游戏、浏览器、其他应用的空间越少。

| 项目 | 单位 | 小白解释 | 怎么看 |
| ---- | ---- | ---- | ---- |
| 内存平均_MB | MB | 测试期间平均占用多少内存 | 越低越轻 |
| 内存峰值_MB | MB | 测试中最高占用多少内存 | 用来看是否有加载峰值 |

### 5.1 内存平均和峰值

| 项目 | 看什么 | 例子 |
| ---- | ---- | ---- |
| 内存平均 | 日常稳定占用 | 空闲长期 180 MB |
| 内存峰值 | 最重的一瞬间 | 切换角色时冲到 600 MB |

### 5.2 内存泄漏

内存泄漏是指应用运行越久，占用越高，而且不降回来。

| 项目 | 小白理解 | 风险 |
| ---- | ---- | ---- |
| 10 分钟 200 MB，2 小时还是 220 MB | 稳定 | 低 |
| 10 分钟 200 MB，2 小时 900 MB | 越跑越胖 | 高 |
| 关闭功能后内存不下降 | 没有释放资源 | 中高 |

对桌宠来说，内存泄漏很危险，因为桌宠可能被用户开一整天。

## 6. GPU 内存指标

GPU 内存也叫显存。它是显卡存放贴图、模型、渲染资源的地方。

| 项目 | 单位 | 小白解释 | 怎么看 |
| ---- | ---- | ---- | ---- |
| GPU内存平均_MB | MB | 平均占用多少显存 | 影响游戏和 3D 应用 |
| GPU内存峰值_MB | MB | 最高占用多少显存 | 用来看模型 / 贴图加载峰值 |

### 6.1 为什么显存会影响游戏

如果桌宠占了不少 GPU 内存，游戏可用的显存就变少。显存紧张时，游戏可能出现：

| 项目 | 说明 |
| ---- | ---- |
| FPS 下降 | GPU 或显存被抢 |
| 画面卡顿 | 资源在显存和内存之间来回搬 |
| 切场景更慢 | 游戏加载资源受影响 |
| 风扇变响 | GPU 长时间更忙 |

## 7. FPS 指标

FPS 是每秒画面帧数。游戏玩家最敏感的通常不是 CPU 数字，而是 FPS 有没有掉。

| 项目 | 单位 | 小白解释 | 怎么看 |
| ---- | ---- | ---- | ---- |
| 游戏基线FPS | FPS | 不开桌宠时游戏帧率 | 作为对照组 |
| 开启桌宠FPS | FPS | 开桌宠后游戏帧率 | 与基线比较 |
| FPS变化_pct | 百分比 | 桌宠导致 FPS 变化多少 | 越接近 0 越好 |

### 7.1 FPS 变化怎么计算

公式：

```text
FPS变化_pct = (开启桌宠FPS - 游戏基线FPS) / 游戏基线FPS * 100
```

例子：

| 项目 | 开启桌宠FPS | FPS变化_pct | 解读 |
| ---- | ---- | ---- | ---- |
| 120 | 119 | -0.8% | 基本无影响 |
| 120 | 114 | -5.0% | 已经可感知 |
| 120 | 96 | -20.0% | 不适合作为默认模式 |

### 7.2 对 desktop-pet 的建议

| 项目 | 说明 |
| ---- | ---- |
| 2D 空闲 | FPS 下降 <= 1% |
| 3D / Live2D 空闲 | FPS 下降 <= 2% |
| 语音 / AI 交互 | 不应造成持续 FPS 下降 |
| 本地 AI / 本地视觉 | 应标记为高资源模式，不默认用于竞技游戏 |

## 8. 网络指标

网络指标用来判断产品是否频繁上传、下载、请求 AI 服务。

| 项目 | 单位 | 小白解释 | 怎么看 |
| ---- | ---- | ---- | ---- |
| 网络流量_MB | MB | 测试场景里用了多少网络流量 | 越高越依赖云端 |

### 8.1 网络指标要和隐私一起看

对 AI 桌宠来说，网络流量不只是性能问题，也是隐私问题。

| 项目 | 说明 |
| ---- | ---- |
| AI 文本聊天 | 是否把聊天发到云端 |
| 语音聊天 | 是否上传音频 |
| 截图 / 视觉 | 是否上传屏幕截图 |
| 角色生成 / 图片生成 | 是否上传图片或提示词 |

## 9. 能耗、功耗、发热指标

能耗和功耗表示设备为了运行这个产品花了多少电。移动端尤其重要，PC 上也会影响风扇和温度。

| 项目 | 单位 | 小白解释 | 怎么看 |
| ---- | ---- | ---- | ---- |
| 能耗或功耗备注 | 文本 | 记录耗电、功耗、发热、风扇等现象 | 没有统一单位时先写观察结论 |

### 9.1 常见记录方式

| 项目 | 说明 |
| ---- | ---- |
| Windows | 功耗估计、风扇、GPU/CPU 长时间占用 |
| macOS | Activity Monitor 的 Energy、powermetrics 输出 |
| Android | batterystats、温度、掉电速度 |
| iOS | Instruments Energy、设备发热、掉电速度 |

### 9.2 小白判断

| 项目 | 说明 |
| ---- | ---- |
| 空闲也发热 | 后台持续计算或渲染 |
| 风扇明显变响 | CPU / GPU 长时间高负载 |
| 手机快速掉电 | 移动端不适合长时间后台陪伴 |
| 语音时耗电升高 | 正常，但需要明确提示用户 |

## 10. 体验指标

性能数字只是工具，最终还是看用户是否觉得“卡、吵、烦、危险”。

| 项目 | 小白解释 | 记录什么 |
| ---- | ---- | ---- |
| 体验备注 | 主观体验记录 | 卡顿、延迟、遮挡、抢焦点、动画是否烦 |
| 请求权限 | 应用请求了哪些系统权限 | 麦克风、屏幕录制、辅助功能、摄像头、通知 |

### 10.1 输入冲突

输入冲突是桌宠影响用户正常操作。

| 项目 | 说明 |
| ---- | ---- |
| 鼠标点到桌宠而不是游戏 | 游戏体验被破坏 |
| 桌宠抢窗口焦点 | 用户输入中断 |
| 覆盖层挡住 UI | 影响操作和信息读取 |
| 全屏游戏中弹窗 | 高打扰 |

### 10.2 权限风险

| 项目 | 说明 |
| ---- | ---- |
| 麦克风 | 可能采集语音 |
| 屏幕录制 | 可能看到游戏、聊天、文件 |
| 辅助功能 | 可能控制点击和键盘 |
| 本地文件 | 可能读取用户文件 |
| 摄像头 | 可能采集人像 |
| 通知 | 可能打扰游戏 |

## 11. 日志和可追溯字段

这些字段用于复查结果，不直接代表性能好坏。

| 项目 | 小白解释 | 为什么重要 |
| ---- | ---- | ---- |
| 源日志路径 | 原始测试日志在哪里 | 方便复查和重新计算 |
| 测试人 | 谁做的测试 | 方便追问测试条件 |
| 日期 | 什么时候测的 | 对齐产品版本和系统版本 |
| 置信度 | 对结果有多相信 | 区分粗略观察和严格测试 |
| 备注 | 其他条件 | 记录异常、账号、网络、设备状态 |

### 11.1 置信度怎么写

| 项目 | 说明 |
| ---- | ---- |
| 高 | 有固定设备、固定场景、完整日志、可重复 |
| 中 | 有日志，但场景或设备控制一般 |
| 低 | 只做了一次观察，或缺少关键日志 |

## 12. 测试场景指标

这些不是 CSV 的单独字段，但在报告和实际测试里经常会用到。

| 项目 | 小白解释 | 常见场景 | 重点看什么 |
| ---- | ---- | ---- | ---- |
| 启动时间 | 从点击到可用花多久 | 冷启动 | 是否慢、是否卡住系统 |
| 磁盘突发 | 启动或加载时读写磁盘 | 冷启动、切角色 | 是否导致系统卡顿 |
| 网络突发 | 短时间大量联网 | 启动、AI 聊天、图片生成 | 是否依赖云端、是否影响延迟 |
| 唤醒次数 | 应用把系统叫醒的频率 | 空闲、后台 | 唤醒多会更耗电 |
| 输入延迟 | 操作到响应的时间 | 拖拽、点击、游戏覆盖层 | 是否跟手 |
| 响应延迟 | 发出请求到得到回答 | AI 文本、语音 | 是否让用户等太久 |
| ASR 延迟 | 语音转文字花多久 | 语音聊天 | 是否影响对话节奏 |
| TTS 延迟 | 文字转语音花多久 | 语音聊天 | 是否说话慢半拍 |
| UI 卡顿 | 界面不顺滑 | 滚动、拖拽、动画 | 是否有掉帧 |
| 帧统计 | 每帧耗时记录 | Android / 游戏场景 | 是否有 jank |
| VRAM | 显存占用 | 3D、游戏覆盖层 | 是否抢游戏资源 |
| 共享内存 | GPU 借用系统内存 | 集成显卡设备 | 是否挤压系统内存 |
| 内存泄漏 | 内存越跑越高 | 长时间稳定性测试 | 是否需要重启 |
| CPU 漂移 | CPU 占用慢慢变高 | 长时间稳定性测试 | 是否后台变重 |
| 句柄 / 线程增长 | 系统资源对象越来越多 | 长时间稳定性测试 | 是否有资源泄漏 |
| 崩溃 / 恢复 | 是否崩、能否恢复 | 长时间稳定性测试 | 是否稳定 |
| 发热 | 设备温度体感或传感器变化 | 移动端、笔记本 | 是否影响长时间使用 |
| 内存压力 | 系统内存紧张程度 | iOS / macOS | 是否挤压其他应用 |

## 13. 一张图式理解

```text
用户觉得卡
  ├─ 可能是 CPU 高：AI、语音、脚本、后台任务太重
  ├─ 可能是 GPU 高：3D、Live2D、透明窗口、游戏抢资源
  ├─ 可能是内存高：模型、贴图、缓存太多
  ├─ 可能是显存高：3D 模型、贴图、WebGPU 占用
  ├─ 可能是网络慢：云端 LLM、语音、图片生成延迟
  └─ 可能是体验冲突：抢焦点、挡 UI、请求权限、弹窗
```

## 14. 怎么读一条测试结果

示例：

| 项目 | 说明 |
| ---- | ---- |
| CPU 平均 | 1.2% |
| CPU P95 | 3.5% |
| CPU 峰值 | 18% |
| GPU 平均 | 2% |
| 内存平均 | 220 MB |
| 游戏基线 FPS | 120 |
| 开启桌宠 FPS | 119 |
| FPS 变化 | -0.8% |

解读：

1. CPU 平均和 P95 都低，说明长期负担小。
2. CPU 峰值 18% 可能是启动或动画突发，只要不频繁就可以接受。
3. GPU 平均 2% 比较轻。
4. 内存 220 MB 对桌宠可接受。
5. FPS 下降不到 1%，对游戏影响很小。

结论：适合作为默认空闲模式。

## 15. desktop-pet 应优先盯住的指标

| 项目 | 指标 | 原因 |
| ---- | ---- | ---- |
| P0 | FPS 变化 | 目标用户是游戏玩家，体感最敏感 |
| P0 | CPU 平均 / P95 | 决定是否长期拖慢系统 |
| P0 | GPU 平均 / P95 | 决定是否抢游戏和桌面合成资源 |
| P0 | 内存平均 / 峰值 | 决定是否能长时间常驻 |
| P0 | 权限请求 | 决定用户信任 |
| P1 | GPU 内存 | 3D / Live2D / 游戏覆盖层必须看 |
| P1 | 能耗 / 发热 | 移动端和笔记本必须看 |
| P1 | 输入延迟 / 抢焦点 | 桌宠体验关键 |
| P2 | 网络流量 | AI / 语音 / 视觉功能上线前重点看 |

## 16. 常见误区

| 项目 | 为什么错 | 正确做法 |
| ---- | ---- | ---- |
| 只看平均 CPU | 平均值会掩盖卡顿尖峰 | 同时看 P95 和峰值 |
| 只看 CPU，不看 GPU | 桌宠可能主要吃显卡 | CPU、GPU、FPS 一起看 |
| 只测 1 分钟 | 看不到内存泄漏和后台漂移 | 空闲至少 10 分钟，稳定性至少 2 小时 |
| 用高配机代表所有用户 | 高配机会掩盖低配问题 | 至少测低配 Windows 和一台常见游戏机 |
| 把系统要求当运行时占用 | 系统要求只是能不能跑 | 必须实际采样 CPU / GPU / 内存 |
| 不记录场景 | 数值无法解释 | 每条数据必须绑定场景 |
| 不记录权限 | 性能和隐私风险分不开 | 权限请求必须写入测试记录 |

## 17. 最小可用测试记录

如果时间很紧，至少记录这些：

| 项目 | 说明 |
| ---- | ---- |
| 产品、版本、平台 | 知道测的是谁 |
| 设备和系统版本 | 知道在哪测 |
| 场景和时长 | 知道测了什么 |
| CPU 平均 / P95 / 峰值 | 看通用计算负担 |
| GPU 平均 / P95 / 峰值 | 看图形负担 |
| 内存平均 / 峰值 | 看常驻成本 |
| 游戏基线 FPS / 开启桌宠 FPS | 看游戏影响 |
| 体验备注 | 记录用户体感 |
| 请求权限 | 记录隐私风险 |
| 源日志路径 | 保证可复查 |

## 18. 这些边界值是怎么来的

先把结论说透：不是所有性能指标都有“权威合格线”。

| 项目 | 有没有权威边界 | 说明 |
| ---- | ---- | ---- |
| 崩溃率、ANR、后台唤醒、后台网络、部分移动端渲染卡顿 | 有，主要来自 Android Vitals / Google Play | Google 会定义 bad behavior threshold，超过会影响商店可见性或触发警告 |
| 启动、卡顿、响应、能耗、内存压力 | 有权威测量体系，但不一定有统一合格线 | Apple MetricKit / Xcode Organizer 会采集这些指标，有时会给推荐值 |
| CPU 平均占用、GPU 平均占用、内存 MB | 没有跨平台统一合格线 | 需要按产品形态、设备档位、是否常驻、是否游戏场景来制定预算 |
| FPS 下降、输入延迟、覆盖层冲突 | 没有单一官方线，但有强行业共识 | 游戏玩家对 FPS 和输入延迟非常敏感，应设更严的产品预算 |

### 18.1 权威指标来源

| 项目 | 权威性 | 覆盖指标 | 可直接拿来用的边界 |
| ---- | ---- | ---- | ---- |
| [Android Vitals](https://developer.android.google.cn/topic/performance/vitals?hl=en) | Google / Android 官方 | 崩溃、ANR、电池、wake lock、启动、慢渲染、低内存杀进程、权限拒绝 | 用户感知崩溃率、用户感知 ANR 率、过度 wake lock 等有 bad behavior threshold |
| [Google Play Console 技术质量](https://support.google.com/googleplay/android-developer/answer/9844486?hl=en) | Google Play 官方 | 后台 wakeup、后台网络、权限拒绝、电池、渲染 | wakeup、后台网络、Wi-Fi 扫描、wake lock 等有具体阈值 |
| [Android Slow Rendering](https://developer.android.google.cn/topic/performance/vitals/render?hl=en) | Android 官方 | 慢帧、冻结帧、ANR | 慢帧 16ms-700ms；冻结帧 700ms-5s；ANR > 5s |
| [Apple Performance and Metrics](https://developer.apple.com/documentation/xcode/performance-and-metrics) | Apple 官方 | 启动、响应、hang、hitch、内存、能耗、磁盘、网络 | 官方提供测量体系；通常按应用和版本看趋势 / 推荐值，不给所有 app 一个固定线 |
| [Apple Understanding Hangs](https://developer.apple.com/documentation/xcode/understanding-hangs-in-your-app) | Apple 官方 | 主线程无响应、hang | Apple 工具通常从主 run loop 无响应超过 250ms 开始报告问题 |
| [Microsoft Windows Performance Analyzer](https://learn.microsoft.com/en-us/previous-versions/windows/desktop/xperf/windows-performance-analyzer--wpa-) | Microsoft 官方 | Windows CPU、GPU、磁盘、线程、调度、系统级性能 | 官方提供分析工具；不规定某个桌面 app CPU/GPU 百分比必须是多少 |
| [RAIL / web.dev](https://web.dev/articles/rail) | Google Chrome 团队 / Web 官方实践 | 响应、动画、空闲、加载 | 用户输入响应目标 100ms；60 FPS 帧预算约 16ms，实践上每帧 JS 工作建议更低 |

### 18.2 Android 上比较“硬”的官方边界

Android 是最适合拿到官方阈值的平台，因为 Google Play 会用 Android Vitals 评估应用技术质量。

| 项目 | 官方边界 / 定义 | 小白理解 |
| ---- | ---- | ---- |
| 用户感知崩溃率 | overall 1.09%；per phone model 8% | 真实用户遇到崩溃的比例不能太高 |
| 用户感知 ANR 率 | overall 0.47%；per phone model 8% | 应用无响应不能频繁发生 |
| 过度部分 wake lock | 5% impacted sessions；beta 指标 | 后台不能长时间偷偷让 CPU 醒着 |
| 慢帧 | 渲染时间 16ms-700ms | 一帧超过 16ms 就可能开始不顺 |
| 冻结帧 | 渲染时间 700ms-5s | 用户会明显感觉卡住 |
| ANR | 大于 5s 无响应 | 应用像死了一样 |
| 过度 wakeup | 每小时超过 10 次 wake-ups 的 impacted sessions | 后台频繁叫醒系统，会耗电 |
| 后台 Wi-Fi 扫描 | 每小时超过 4 次后台 Wi-Fi scans | 后台扫描太频繁会耗电 |
| 后台网络 | 后台每天超过 50 MB 移动网络流量 | 用户没看见时不该大量跑流量 |
| watch face 电池 | 超过 4.44% / hour | 表盘类应用不应快速耗电 |

这些适合移动端 companion app，不能直接照搬到 Windows 桌宠，但可以作为“后台常驻应用不应打扰系统”的参考。

### 18.3 Apple 上的权威口径

Apple 更强调“用官方工具测趋势和真实用户体验”，不是给所有 app 一个固定 CPU 合格线。

| 项目 | Apple 口径 | 对 desktop-pet 的用法 |
| ---- | ---- | ---- |
| 启动时间 | Xcode Organizer / MetricKit 看 time to first frame | 冷启动记录到首帧时间 |
| 响应性 | 看 hang、hitch、UI responsiveness | 拖拽、点击、悬停不能卡 |
| hang | 主 run loop 长时间无响应，Apple 工具通常从 250ms 以上报告 | 桌宠主线程不能被 AI / IO 堵住 |
| 内存 | 看 memory use、memory pressure、termination | 长时间常驻不能越跑越胖 |
| 能耗 | 看 battery / energy impact | 空闲桌宠不能让笔记本明显耗电 |
| 磁盘写入 | 看 disk writes | 记忆、日志、缓存不要频繁写盘 |

所以 Apple 这边的“权威”更像评测框架：用官方工具采集启动、响应、hang、内存、能耗，再和上一版本、同类场景、设备档位比较。

### 18.4 Windows / macOS 桌面端为什么没有统一 CPU 百分比线

桌面端很难有一个权威线，是因为：

1. CPU 百分比的含义会随核心数、采集工具、系统口径变化。
2. GPU 百分比会随驱动、GPU 引擎、集成 / 独立显卡不同而变化。
3. 内存占用要看设备总内存、是否 Electron / Tauri / 原生、是否加载 3D 模型。
4. 桌宠是常驻应用，同样 3% CPU，开 10 秒和开 8 小时的意义完全不同。
5. 游戏场景更敏感，FPS 下降比 CPU 数字更重要。

因此 Windows / macOS 更常见的权威做法是“权威工具 + 自定义预算”：

| 项目 | 权威工具 | 我们自己定义的预算 |
| ---- | ---- | ---- |
| Windows | Performance Monitor、Windows Performance Analyzer、Task Manager GPU counters | CPU / GPU / 内存 / FPS 下降目标 |
| macOS | Activity Monitor、Instruments、powermetrics、MetricKit | CPU / GPU / 内存 / 能耗 / hang 目标 |

### 18.5 desktop-pet 当前边界值属于什么

本报告里的这些目标：

| 项目 | CPU | GPU | 内存 | FPS 影响 |
| ---- | ---- | ---- | ---- | ---- |
| 2D 空闲 | 平均 <= 1%，P95 <= 3% | 平均 <= 2% | <= 250 MB | FPS 下降 <= 1% |
| 3D / Live2D 空闲 | 平均 <= 2%，P95 <= 5% | 平均 <= 5% | <= 400 MB | FPS 下降 <= 2% |
| 语音模式 | 平均 <= 10%，P95 <= 30% | <= 8% | <= 800 MB | 不冲突游戏音频 |

不是官方标准，而是 desktop-pet 的产品预算。它来自 4 个判断：

1. 桌宠是常驻应用，空闲时必须非常轻。
2. 目标用户是游戏玩家，FPS 下降比普通工具软件更敏感。
3. AI / 语音 / 视觉是高资源能力，不能默认开启。
4. 竞品公开资料大多不给实测值，所以先用保守预算保护体验。

更准确的说法应该是：

| 项目 | 例子 | 应该怎么写 |
| ---- | ---- | ---- |
| 官方阈值 | Android ANR 0.47%、慢帧 16ms-700ms | “官方 bad behavior threshold / 官方定义” |
| 官方测量体系 | Apple MetricKit、Windows WPA | “官方推荐的测量指标 / 工具” |
| 产品预算 | desktop-pet 2D 空闲 CPU <= 1% | “内部目标，需要后续用实测校准” |
| 行业经验 | FPS 下降 <= 1%-2% | “游戏场景保守体验目标” |

### 18.6 后续应该如何校准边界

等有真实测试数据后，不应该死守第一版边界，而要按下面方式校准：

| 项目 | 做什么 | 输出 |
| ---- | ---- | ---- |
| 1 | 测 P0 竞品空闲、交互、游戏覆盖层 | 竞品基线 |
| 2 | 测 desktop-pet 原型相同场景 | 自身基线 |
| 3 | 找低配 Windows 和主流游戏场景 | 最敏感设备线 |
| 4 | 用用户体感验证 FPS / 输入延迟 | 体验阈值 |
| 5 | 把阈值分成默认模式和高资源模式 | 产品性能 SLA |

建议最后形成三档：

| 项目 | 含义 | 处理 |
| ---- | ---- | ---- |
| 绿色 | 默认可接受 | 可以默认开启 |
| 黄色 | 有感知风险 | 需要提示或自动降级 |
| 红色 | 不适合默认 | 必须用户主动开启，或禁用于游戏场景 |
