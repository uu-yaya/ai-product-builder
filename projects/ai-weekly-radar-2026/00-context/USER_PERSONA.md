## 2. Secondary Personas

本项目为内部研究项目，不存在外部次要用户。可被视为"消费方"的内部对象：

- **APB 其他工作区 / 线程**（如 PM Strategy / Engineering Build）：将周报作为立项与方案评审的输入。
- **未来回查者（同一作者本人）**：在 4–8 周后回查周报历史寻找决策证据。

注：以上为"消费方"而非传统人格，不再单独画像。

## 3. User Goals

主要用户希望通过本项目达成的目标（基于 [PROJECT_CONTEXT.md §5](PROJECT_CONTEXT.md)）：

- 每周稳定获得一份 AI Agent / AI Coding / 游戏 AI 三方向的进展摘要。
- 单期阅读耗时 ≤10 分钟即能形成判断。
- 立项 / 方案评审时能快速回查 4–8 周相关条目作为证据。
- 区分"事实 / 观点 / 推测"，避免被噪声误导。
- 长期沉淀可检索的研究资产，而不是阅后即焚的信息流。

## 4. Pain Points

- **信息不透明**：AI 领域日更新量大，原始信源分散在 19+ 个渠道。
- **低频高代价**：错过一个真正的拐点（例如新 Agent 范式、新 Coding 工具、新 Game AI 商业化案例），代价高。
- **流程问题**：单点新闻无法直接进入决策；需要被结构化、被分级、被串联。
- **工具缺失**：现有 AI Newsletter / 行业聚合站偏大众科普，与"产品 + 工程"决策视角错位。
- **高频低代价**：每天看零散新闻浪费时间但单次没有明显损失，需要刻意收口为周更。

## 5. Motivations

- 效率：用 ≤2 小时 / 周获得三方向的判断输入。
- 决策赋能：把周报作为"立项 / 方案 / 引入工具"的证据库。
- 知识资产：周报历史本身具有复利价值（跨期串联）。
- 信任：自己掌握信源池与可信度规则，不依赖外部内容方的取舍。

## 6. Usage Context

- **写作时点**：每周一上午（节假日可顺延）。
- **写作工具**：本仓库 + 浏览器（一级 / 二级信源）+ 临时检索三级源。
- **阅读时点**：每周一阅读当周；立项 / 方案评审时回查历史。
- **协作**：单人项目；不涉及外部协作。
- **状态恢复**：周报本身就是状态恢复的载体，跨期回看即可重新进入上下文。

## 7. Decision Factors

用户判断"周报值得继续做"的因素：

- **输出质量**：三大方向是否真的捕捉到了拐点，还是只是新闻搬运。
- **速度**：单期收口耗时是否 ≤2 小时；阅读耗时是否 ≤10 分钟。
- **可控性**：信源池、字段、可信度规则是否始终在自己手中。
- **可检索性**：是否能在 4–8 周后快速回查到具体条目。
- **学习成本**：模板与流程是否稳定，避免每期重新设计。
- **信任**：是否区分了事实 / 观点 / 推测；中文源是否被正确降级。

## 8. Success Criteria

从用户视角，"用得好"的判断标准（口语化）：

- "这一期我看完了，知道下周该关注什么。"
- "这一条让我立刻想起一个产品机会。"
- "这一期帮我否决了一个不靠谱的立项。"
- "回看上个月的周报，能直接定位当时的判断依据。"
- "我没把社区八卦写成事实。"
