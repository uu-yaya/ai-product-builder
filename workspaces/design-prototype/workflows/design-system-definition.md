# Design System Definition Workflow

## 1. Purpose

Define a lightweight design system that supports consistent visual execution and engineering handoff.

## 2. When to Use

Use when a project needs reusable visual rules, component standards, interaction states, or tokens before prototype or implementation.

## 3. Inputs

- Brand keywords
- Product type
- Platform
- Target users
- Visual references or direction
- Existing UI constraints

## 4. Steps

1. Define visual keywords and design principles.
2. Define color tokens.
3. Define typography tokens.
4. Define spacing, radius, and shadow tokens.
5. Define icon style.
6. Define component rules and variants.
7. Define state rules and motion guidelines.
8. Define accessibility rules.
9. Add engineering implementation notes.

## 5. Outputs

- 视觉关键词
- 色彩 token
- 字体 token
- 间距 token
- 圆角 token
- 阴影 token
- 图标风格
- 组件规范
- 状态规范
- 动效规范
- 可访问性规范
- 工程实现注意事项

## 6. Quality Checklist

- Tokens use stable English names.
- Component states are included.
- Accessibility rules are explicit.
- Design system fits the current MVP.
- Engineering notes are practical.

## 7. Common Mistakes

- Creating too many tokens before the product stabilizes.
- Missing state rules.
- Ignoring accessibility.
- Defining visual rules that engineering cannot implement.
- Copying another product's design system directly.
