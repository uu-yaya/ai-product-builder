# Requirement Clarification Workflow

## 1. Purpose

Clarify ambiguous product requests before writing PRD, designing UI, or planning engineering work.

## 2. When to Use

Use when the request is incomplete, too broad, solution-first, technically unclear, or missing user context.

## 3. Inputs

- Original request
- Known user or customer context
- Business goal
- Constraints
- Existing product or project context

## 4. Steps

1. Restate the request in clear language.
2. List known information.
3. Identify missing information.
4. Ask clarification questions grouped by user, scenario, value, scope, constraints, success metrics, and risks.
5. Propose safe assumptions that allow progress without inventing facts.
6. Define a draft MVP scope.
7. Identify risks and next steps.

## 5. Outputs

- 需求复述
- 已知信息
- 缺失信息
- 澄清问题
- 可先行假设
- MVP 范围
- 风险
- 下一步建议

## 6. Quality Checklist

- User, scenario, pain point, goal, value, MVP, and risk are covered.
- Assumptions are separated from facts.
- Missing information is explicit.
- Questions are actionable and prioritized.
- MVP scope is small enough to validate.

## 7. Common Mistakes

- Jumping directly into implementation.
- Asking too many unfocused questions.
- Treating assumptions as facts.
- Expanding scope before validating the core problem.

## 8. Example Command

请进入 pm-strategy 工作区，帮我澄清这个需求，并输出已知信息、缺失信息、澄清问题、MVP 范围和风险。
