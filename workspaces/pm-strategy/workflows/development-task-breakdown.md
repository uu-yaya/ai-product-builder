# Development Task Breakdown Workflow

## 1. Purpose

Turn a PRD or clarified requirement into engineering-ready tasks grouped by owner role.

## 2. When to Use

Use after the PRD has enough clarity to support design, implementation, QA, and launch planning.

## 3. Inputs

- PRD or requirement summary
- MVP scope
- User flow
- Functional requirements
- AI requirements, if any
- Constraints and dependencies

## 4. Steps

1. Read the PRD and identify modules.
2. Separate tasks by Product, Design, Frontend, Backend, AI / Algorithm, QA, and DevOps.
3. For each task, define Task, Owner Role, Priority, Input, Output, Dependencies, Acceptance Criteria, and Risks.
4. Identify cross-role dependencies.
5. Cut or defer non-MVP items.
6. Produce a task table and next implementation sequence.

## 6. Quality Checklist

- All required owner roles are considered.
- Each task has a clear input and output.
- Acceptance criteria are testable.
- Dependencies are explicit.
- MVP cuts are visible.

## 7. Common Mistakes

- Mixing product decisions with implementation tasks.
- Missing QA or DevOps tasks.
- Leaving AI evaluation undefined.
- Writing tasks without acceptance criteria.
- Overloading the first milestone.
